iContractBot - Your guide for creating smart contracts

Important concepts and instructions

First of all, thank you for using iContractBot. This chtabot will help you to generate smart contracts. It would be great if you could go through all the instructions.


Points to be noted

  • Please try to be elaborate with the chatbot. For example, if you want to create a participant, then the input should be Create a participant or I want to create a participant. The actions that have to be carried out are closely knit. Therefore, if enough information is not provided, then the chatbot may generate wrong codes.
  • When you are dealing with a particular element, then specify that you intend to act upon it. For example, suppose you want to set the identifer for a participant, then the input should be I want {name} to be the identifier for the participant.
  • If the chatbot replies Sorry, I didn't get you, then don't get panicked. It's because the chatbot got confused between two states. Therefore, as mentioned in the first point, try to be a little more elaborate.
  • The terms participant and identifier have been explained below for clarity.
  • Also, there's a use case, which hopefully, will make things clearer.
  • Next up, there's a test case, which you have to generate using the chatbot.
  • Please go through the sections given below for more clarity. This would make the generating smart contract for the test case easier.
  • We have kept the test case as simple as possible where you have to only create elements. So, take it easy.
  • This is what you are going to create in the whole process.
  • Contract contains everything that can be imposed onto a contract.
  • A contract is made up of three basic elements - Participant, Asset and Transaction. Therefore, you will be creating/editing/deleting these three while interacting with the chatbot.
  • When writing a smart contract one would have to define a name for it and the targeted platform (Hyperledger Composer, Microsoft Azure, Ethereum).

Instructions while chatting with the bot

  • You'll have to say that you want to create a contract to start the conversation. Then, the name has to be specified.
  • Then, you keep adding or editing elements to the contract.
  • After you are done adding all the approproate elements to the chatbot, just say The contract is complete.
  • Please edit the bot according to the hierarchy. That is, the building blocks of a contract are elements - participants, assets and transactions. Therefore, first create them. The building blocks of elements are parameters and relationships. Therefore, you first create or edit an element and then, you create/edit/delete parameters or relationship.
  • Assets are tangible or intangible goods, services, or property.
  • Assets can represent almost anything of value, whether physical or virtual, for example, a house for sale, the sale listing, the land registry certificate for that house, and the insurance documents for that house may all be assets in one or more smart contracts.
  • Assest contain parameters which characterise them.
  • Assets must have a unique identifier, but other than that, they can contain whatever parameters you define.
  • Assets may be related to a transaction in an AssetRel relationship.
  • Participants are members of a smart contract. They are the ones who'll be using the functionalities of the smart contract.
  • They may own assets and submit transactions.
  • Participants contain parameters which characterise them.
  • Participant types are modeled, and like assets, must have an identifier and can have any other parameters as required.
  • In blockchain scenario, transactions represent a means for executing some task.
  • Transactions are the mechanism by which participants interact with assets.
  • This could be as simple as a participant placing a bid on an asset in an auction, or an auctioneer marking an auction closed, automatically transferring ownership of the asset to the highest bidder.
  • Parameters (attributes) represent a user-defined type to be included in elements such as asset, participant and transactions.

    • Relationship represents the relationships that different elements can have. They can contain Events with their own description and a name. These relationships can come in two types:

    • The first one is TranRel, which is a relationship between Participant and Transaction.
    • The second one is AssetRel, which is a relationship between Transaction and Asset.

    Lets consider vehicle auction as an example. In this use case, a smart contract is used to auction vehicles, where the vehicle is the key asset. There are two participants in this example: owner and bidder. The bidder can place bids on the vehicles that they are interested in. So in summary we have:

    • Asset: vehicle
    • Participant: owner and bidder
    • Transaction: place bid
